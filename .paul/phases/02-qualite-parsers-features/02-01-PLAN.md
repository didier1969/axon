<objective>
## Goal
Add byte offsets to sql/yaml regex parsers and implement the `axon_read_symbol` MCP tool that returns exact source code via O(1) file slice.

## Purpose
`sql_lang.py` and `yaml_lang.py` currently emit `start_byte=0, end_byte=0` (as noted in v0.6 decision: "regex-based parsers have no tree-sitter node"). This leaves SQL and YAML symbols unreadable by `axon_read_symbol`. Fixing byte offsets in these two parsers unblocks the new tool and completes the byte-offset story across all 12 languages.

The `axon_read_symbol` tool is the highest-value feature of Phase 2: it lets AI agents retrieve exact symbol source in one call with no token waste (file[start_byte:end_byte] instead of scanning the whole file).

## Output
- `sql_lang.py` symbols carry correct `start_byte` / `end_byte`
- `yaml_lang.py` / toml symbols carry correct `start_byte` / `end_byte`
- New `axon_read_symbol` MCP tool registered and dispatched
- All 852+ tests still pass; 3 new tests added
</objective>

<acceptance_criteria>

## AC-1: SQL byte offsets
```gherkin
Given a .sql file with CREATE TABLE, CREATE VIEW, CREATE FUNCTION, CREATE PROCEDURE symbols
When SqlParser.parse() is called
Then each SymbolInfo has start_byte = byte offset of the CREATE keyword match in the content string
  And end_byte = byte offset of the semicolon ending the statement + 1
  And start_byte > 0 for any non-first symbol
```

## AC-2: YAML/TOML byte offsets
```gherkin
Given a .yaml or .toml file with top-level and nested keys
When YamlParser.parse() is called
Then each SymbolInfo has start_byte = byte position of the start of the matching line (UTF-8)
  And end_byte = start_byte + byte length of that line (including newline)
  And start_byte > 0 for any non-first-line symbol
```

## AC-3: axon_read_symbol tool — happy path
```gherkin
Given a repo is indexed and a symbol exists with valid start_byte/end_byte
When axon_read_symbol is called with the symbol name (and optionally file/repo)
Then the tool returns the exact source content: file_bytes[start_byte:end_byte].decode("utf-8", errors="replace")
  And the response includes the file path and symbol name as header
```

## AC-4: axon_read_symbol tool — fallback
```gherkin
Given a symbol has start_byte=0 and end_byte=0 (legacy or regex-parser symbol)
When axon_read_symbol is called
Then the tool returns the stored content field from the graph node instead
  And a note "(byte offsets unavailable, using stored content)" is included
```

## AC-5: axon_read_symbol tool — not found
```gherkin
Given no symbol matches the given name (and optional file/repo filter)
When axon_read_symbol is called
Then the tool returns "Symbol not found: {name}"
```

  <action>
    For each regex match `m` (CREATE TABLE, VIEW, FUNCTION, PROCEDURE), compute:
    - `start_byte = m.start()` — byte offset of the match start in the content string
    - `end_byte`: scan forward from `m.start()` for the first `;` character, then `end_byte = content.index(';', m.start()) + 1`. If no `;` found, use `len(content.encode('utf-8'))` — but since content is Python str and bytes may differ for non-ASCII, use `len(content[:content.find(';', m.start()) + 1].encode('utf-8'))` for accuracy. Simpler: since SQL files are almost always ASCII, `end_byte = content.find(';', m.start()) + 1` (character offset = byte offset for ASCII). If `content.find(';', m.start()) == -1`, use `len(content)`.
    - Pass `start_byte=start_byte, end_byte=end_byte` to each `SymbolInfo(...)` constructor call.
    - For DROP/ALTER calls (CallInfo), no byte offset needed (CallInfo has no byte fields).

    Avoid: changing any existing field values (name, kind, start_line, end_line, content).

  <name>Task 2: Add byte offsets to YamlParser</name>
  <files>src/axon/core/parsers/yaml_lang.py</files>
  <action>
    Pre-compute a list of line start byte offsets before the parsing loop:
    ```python
    # Compute cumulative byte offsets for each line (UTF-8)
    encoded_lines = [line.encode("utf-8") for line in content.splitlines(keepends=True)]
    line_start_bytes: list[int] = [0]
    for el in encoded_lines:
        line_start_bytes.append(line_start_bytes[-1] + len(el))
    ```
    Then for each symbol at `line_no` (1-indexed):
    - `start_byte = line_start_bytes[line_no - 1]`
    - `end_byte = line_start_bytes[line_no]` (start of next line = end of this line incl. newline)
    - If `line_no >= len(line_start_bytes)`: `end_byte = len(content.encode("utf-8"))`

    Apply the same logic in both `_parse_yaml()` and `_parse_toml()`.

    Both methods currently receive `content` as their first arg but not the pre-computed offsets.
    Pass `line_start_bytes` as a third argument to both private methods, or compute it inline.

    Avoid: changing the symbol names, kinds, or line numbers.
  </action>
