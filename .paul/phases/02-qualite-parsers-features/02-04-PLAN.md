<objective>
## Goal
Three focused improvements: (1) replace `recv(4096)` loop with `makefile("rb").readline()` in the MCP socket client, (2) append `[BATCH WARNING: N/M failed: indices [...]]` when axon_batch has partial failures, (3) read `AXON_LRU_SIZE` env var as the default for `--max-dbs` in the daemon.

## Purpose
- **readline()**: eliminates a subtle buffer fragmentation bug — `recv(4096)` may not read the full response in one call if the response is exactly 4096 bytes; `readline()` reads until `\n` correctly and is the idiomatic approach for newline-framed protocols.
- **Partial failure summary**: makes batch failures visible. Currently a failing sub-call is silently mixed into the result; the warning lets callers detect and act on partial failures.
- **AXON_LRU_SIZE**: allows users to tune the daemon cache size without rebuilding — useful for machines with more RAM or repos that need more backends cached.

## Output
- `src/axon/mcp/server.py` updated (`_try_daemon_call`, `_batch_daemon_call`, direct fallback in `call_tool`)
- `src/axon/daemon/__main__.py` reads `AXON_LRU_SIZE` env var
- Tests added for all three behaviors
</objective>

  <name>Task 1: Replace recv loop with makefile readline in socket client</name>
  <files>src/axon/mcp/server.py</files>
  <action>
    In `_try_daemon_call` (around line 124-130), replace:
    ```python
    data = b""
    while not data.endswith(b"\n"):
        chunk = sock.recv(4096)
        if not chunk:
            break
        data += chunk
    ```
    with:
    ```python
    data = sock.makefile("rb").readline()
    ```
    The response protocol is newline-framed (each response is one JSON line + `\n`),
    so `readline()` is the correct and idiomatic read.

    In `_batch_daemon_call` (around line 165-170), replace the same recv pattern inside
    the per-call loop with `data = sock.makefile("rb").readline()`.

    Do NOT change the encoding/decoding, the error handling, or anything else in these functions.
    Avoid: creating the makefile outside the `with sock:` block (it would close on socket close).
    Note: `sock.makefile("rb")` does NOT close the socket when the file object goes out of scope
    (the socket remains open for subsequent calls in the batch loop). This is correct behaviour.
  </action>
