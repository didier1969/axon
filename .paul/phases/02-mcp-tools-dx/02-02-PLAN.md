## Goal
Add `axon_lint` MCP tool that detects high-coupling symbols, god classes, and circular import cycles in the knowledge graph. Fix the `cohesion: 0.0` placeholder in community detection by computing real modularity scores using igraph.

## Purpose
`axon_lint` turns axon into an actionable code quality tool — AI agents can identify structural anti-patterns without writing Cypher queries. The cohesion metric has been a placeholder since v0.5; replacing it with real modularity makes community data meaningful for quality assessments and future features (e.g. axon_summarize grouping by community).

  <name>Task 1: Implement handle_lint in tools.py</name>
  <files>src/axon/mcp/tools.py</files>
  <action>
    Add `handle_lint(storage, repo=None)` after `handle_find_usages` (or after `handle_dead_code`).

    Pattern: same try/finally repo-loading pattern as other handlers. No symbol arg needed (lint scans everything).

    Three lint rules implemented via execute_raw with parameterized queries:

    **Rule 1 — High coupling (fan-out > 20):**
    ```python
    HIGH_COUPLING_THRESHOLD = 20
    rows = storage.execute_raw(
        "MATCH (n)-[r:CodeRelation]->(callee) "
        "WHERE r.rel_type = 'calls' "
        "RETURN n.name, n.file_path, count(callee) AS out_degree "
        "ORDER BY out_degree DESC LIMIT 20"
    ) or []
    high_coupling = [(row[0], row[1], int(row[2])) for row in rows if row[2] and int(row[2]) > HIGH_COUPLING_THRESHOLD]
    ```

    **Rule 2 — God classes (> 15 methods):**
    ```python
    GOD_CLASS_THRESHOLD = 15
    rows = storage.execute_raw(
        "MATCH (cls:Class)-[r:CodeRelation]->(m:Method) "
        "WHERE r.rel_type = 'defines' "
        "RETURN cls.name, cls.file_path, count(m) AS method_count "
        "ORDER BY method_count DESC LIMIT 20"
    ) or []
    god_classes = [(row[0], row[1], int(row[2])) for row in rows if row[2] and int(row[2]) > GOD_CLASS_THRESHOLD]
    ```

    **Rule 3 — Import cycles (2-cycles only):**
    ```python
    rows = storage.execute_raw(
        "MATCH (a)-[r1:CodeRelation]->(b)-[r2:CodeRelation]->(a) "
        "WHERE r1.rel_type = 'imports' AND r2.rel_type = 'imports' "
        "AND a.file_path < b.file_path "
        "RETURN a.file_path, b.file_path LIMIT 20"
    ) or []
    import_cycles = [(row[0], row[1]) for row in rows if row[0] and row[1]]
    ```
    Note: `a.file_path < b.file_path` deduplicates A↔B vs B↔A pairs (returns each pair once).

    Format output:
    ```
    # Lint Report

    ## High Coupling (fan-out > 20) — N issues
    1. symbol_name  src/file.py  [out-degree: 25]
    ...

    ## God Classes (> 15 methods) — N issues
    1. MyClass  src/models.py  [methods: 18]
    ...

    ## Import Cycles — N issues
    1. src/a.py  ↔  src/b.py
    ...

    Tip: High coupling → consider splitting into smaller functions.
    Tip: God classes → extract cohesive groups into separate classes.
    Tip: Import cycles → introduce an interface or shared utility module.
    ```

    If all three lists are empty: return "No structural issues detected. Codebase looks healthy."

    Avoid: 3-cycles or deeper (too expensive, deferred to v0.9). Avoid: hardcoding table names in Cypher — use the generic `CodeRelation` relationship type that the existing schema uses.

  <name>Task 3: Fix community cohesion — compute real modularity score</name>
  <files>src/axon/core/ingestion/community.py</files>
  <action>
    In `process_communities`, replace the `"cohesion": 0.0` placeholder with a real modularity calculation.

    The modularity must be computed per-community relative to the full graph partition. The approach:

    1. After all communities are detected (after `all_member_groups` is built), compute the full-graph partition membership list required by igraph.modularity():
       ```python
       # Build membership list: all_member_groups is a list of groups (each group = list of vertex indices)
       # We need a flat list where membership[vertex_idx] = community_idx
       membership: list[int] = [-1] * ig_graph.vcount()
       for comm_idx, group in enumerate(all_member_groups):
           for vertex_idx in group:
               membership[vertex_idx] = comm_idx
       # Vertices not in any community (singletons filtered out) get their own singleton group
       singleton_idx = len(all_member_groups)
       for i, m in enumerate(membership):
           if m == -1:
               membership[i] = singleton_idx
               singleton_idx += 1
       ```

    2. Compute the global modularity (one value for the whole partition):
       ```python
       try:
           global_modularity = ig_graph.modularity(membership)
       except Exception:
           global_modularity = 0.0
       ```

    3. For each individual community, compute a local cohesion score as:
       - (edges inside community) / (total edges in full graph) if total edges > 0, else 0.0
       - This is a normalized intra-community density, bounded [0.0, 1.0]
       ```python
       total_edges = ig_graph.ecount()
       # For community i with original_member_ids (vertex indices):
       sub = ig_graph.subgraph(original_member_ids)
       intra_edges = sub.ecount()
       cohesion = intra_edges / total_edges if total_edges > 0 else 0.0
       ```

    4. Store `cohesion` in the community node (replace the hardcoded 0.0):
       ```python
       community_node = GraphNode(
           ...
           properties={
               "cohesion": round(cohesion, 4),
               "symbol_count": len(member_ids),
               "modularity": round(global_modularity, 4),
           },
       )
       ```

    The `ig_graph` and `all_member_groups` are already available in scope at the point where community nodes are created (the loop at line ~198). The subgraph computation is cheap because igraph subgraph() operates on already-built vertex index lists.

    Avoid: calling leidenalg again — use the already-computed partition. Avoid: modifying the GraphNode schema (properties dict is already a free-form dict). Avoid: changing the community detection algorithm or its parameters.

    Important: the `subgraph` call goes inside the community creation loop, computed once per community — not once for the whole graph. This ensures correct per-community values.
  </action>
