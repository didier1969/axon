UNE source canonique par classe de métrique : (a) métriques de stock par-projet (files/symbols/chunks/embeddings/edges) servies par la vue PG unique ist.project_telemetry, déjà ventilée par project_code ; (b) métriques de flux haute-fréquence (rates, workers, queues, scheduler) publiées par l'indexer (propriétaire) à chaque tick dans une ligne PG axon_runtime.indexer_runtime_truth, le brain n'étant que lecteur pur. Un compositeur unique brain-side assemble l'enveloppe dashboard_state_v1 en lisant ces deux sources canoniques, puis fait un fan-out IDENTIQUE vers les trois consommateurs : humains/dashboard (socket télémétrie → PubSub → LiveViews), LLM/MCP (status, project_status, embedding_status, diagnose_indexing, mcp_surface_diagnostics projettent la même enveloppe), automation/CLI (axon-live status). Aucune requête ad-hoc ni pont fichier : truth observable identiquement partout, refresh automatique des trois surfaces (REQ-AXO-309).

Faits vérifiés (MCP / code)

  • La vue canonique ist.project_telemetry existe (db/ddl/03_ist_schema.sql:329) et expose par project_code : files_total/discovered/indexed/failed/skipped, symbols, chunks_total/embedded/pending, edges. Aucun consommateur Rust ne la lit (grep project_telemetry dans src/axon-core/src = 0).
  • Le dashboard utilise UN seul événement (dashboard_state_v1) — bon design — composé brain-side par compose_dashboard_state_v1 (dashboard_state.rs:253) à 1 Hz, mais à partir de 3 sources hétérogènes : composite PG dashboard_state_full(3), cached_fs_counters() global, métriques RAM + pont fichier.
  • dashboard_state_full(3) (db/ddl/08:211) ne lit PAS la vue : il appelle dashboard_per_project_counts (db/ddl/08:39) qui recompute un rollup ad-hoc (UNION DISTINCT project_code + 4 LEFT JOIN sur ist.symbol/chunk/edge/chunkembedding) et déclare explicitement (commentaire 57-62) que les file-counts par projet sont impossibles depuis ce chemin.
  • embedding_status (tools_system.rs:303) refait ses propres requêtes scalar sur ist.Chunk/ChunkEmbedding/Symbol/IndexedFile/Edge/Project + un per-project breakdown SQL maison, et appelle le même cached_fs_counters() global — mélangeant disk/eligible GLOBAL avec symbols/chunks scoped (scope-honesty cassée, cible REQ-AXO-901865).
  • diagnose_indexing (tools_governance.rs:328 → indexing_diagnosis_markdown:47 + file_count_for_project:334) constitue un TROISIÈME jeu de requêtes ad-hoc (count(*) FROM ist.IndexedFile, count(DISTINCT file_path) FROM ist.Chunk).
  • cached_fs_counters() (tools_system.rs:43-83) fait un ignore::WalkBuilder complet de AXON_WATCH_DIR à chaque expiration TTL 60s et retourne un couple (disk_files, eligible_files) GLOBAL non ventilé par projet — la cible REQ-AXO-901749 est un DashMap<project_code, ProjectFileCounters> incrémental rempli au boot + ajusté par inotify.
  • Les métriques de flux (rates/graph_workers/queues) arrivent via le pont FICHIER runtime-heartbeat.json lu par split_peer_runtime_info (runtime_topology_support.rs:215, main_telemetry.rs:229) + snapshot RAM brain-local ; REQ-AXO-901854 cible une ligne PG axon_runtime.indexer_runtime_truth écrite par l'indexer (propriétaire), brain lecteur pur, retrait du pont fichier (delta observabilité ~72%->90%).
  • Côté Elixir le fan-out est propre : socket /tmp/axon-telemetry.sock → BridgeClient (line-buffered, topic dédié dashboard:state, REQ-AXO-901826) → DashboardState.from_map → Phoenix.PubSub.local_broadcast → LiveViews (pipeline_live/projects_live/mcp_live).
  • PIL-AXO-001 acceptance exige que status() et project_status() ne divergent jamais sur HEALTHY/DEGRADED et que CLI/MCP émettent des champs d'évidence identiques ; aujourd'hui project_status (tools_framework_rationale.rs:13) délègue à axon_status sans projeter la vue de métriques, donc divergence possible sur les chiffres de stock.
  • REQ-AXO-309 (planned) : les 3 surfaces (autodoc SOLL, dashboard, Memgraph) sont rafraîchies à la demande uniquement, sans déclencheur automatique — risque de dérive silencieuse vs la vérité PG.
