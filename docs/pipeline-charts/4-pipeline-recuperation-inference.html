DOC-4-RETRIEVAL · Axon · idéal vs actuel vs delta · 2026-06-05 (post-session 73)

1 · Flux idéal SOTA (la cible, raison d'être d'Axon)

C'est l'ÂME d'Axon (VIS-AXO-001, lignée Peirce→Wright→Pearl) et non un meilleur moteur de recherche. Un agent — LLM ou humain — pose une question ; aujourd'hui il devine l'intention en lisant la source (ABDUCTION FAILLIBLE au sens de Peirce : une inférence vers la meilleure explication, structurellement révisable, donc une source d'erreur). Axon SUPPRIME cette abduction en FUSIONNANT, AU POINT DE RÉCUPÉRATION, le QUOI (la structure : IST) ET le POURQUOI (l'intention : SOLL), pour rendre une réponse correcte PAR CONSTRUCTION — l'agent cesse de viser la mauvaise cible. Cinq principes premiers : (1) FUSION TRI-MODALE DU QUOI — un même besoin est éclairé par trois lentilles complémentaires, FTS lexical (le terme exact) + vecteur sémantique (pgvector HNSW, embeddings BGE-Large 1024d, le sens proche) + graphe structurel (snapshot RAM CSR IstGraphView, les relations réelles : appels, imports, blast radius), recombinées par Reciprocal Rank Fusion (RRF) en un classement unique robuste qu'aucune lentille seule n'atteint. (2) FUSION DU POURQUOI — le SOLL gouvernant (vision / piliers / décisions / exigences) est JOINT au quoi à la récupération : why-primary, how-inferred, l'intention est primaire et le comment s'en infère, asymétrie irréductible (le pourquoi élague le comment ; le comment ne reconstitue le pourquoi qu'avec perte). (3) SURFACE AGENT-NATIVE UNIQUE (PIL-AXO-002) — la sortie est machine-actionnable : next_action explicite, recovery hints, enveloppe structurée, ZÉRO lecture de source par l'agent. (4) CALIBRATION HONNÊTE DE LA CONFIANCE — fraîcheur et trust sont exposés et le fallback est un GRADIENT explicite (CPT-AXO-029 : freshness = trust calibration, PAS un gate ; un read reste utilisable même quand l'indexer est idle, on cross-vérifie au lieu de refuser). (5) RÉPONSE CORRECTE-PAR-CONSTRUCTION — le résultat porte à la fois le code pertinent ET son intention gouvernante, de sorte que l'agent n'a plus rien à deviner. Le flux idéal : Question agent → surface MCP agent-native → fusion tri-modale RRF (FTS + vecteur + graphe) ⊕ fusion SOLL (le pourquoi gouvernant) → réponse correcte-par-construction (quoi + pourquoi + next_action).

Pourquoi c'est l'idéal (pas un refactor) : Ce n'est pas un meilleur moteur de recherche, et donc pas un refactor de la récupération existante : c'est la SUPPRESSION STRUCTURELLE de l'abduction faillible. Dans toute approche RAG/grep classique, l'agent récupère du code puis INFÈRE l'intention par abduction — au sens strict de Peirce, une inférence vers la meilleure explication, faillible et révisable par essence ; c'est cette étape devinette qui produit les mauvaises cibles, les régressions « ça marchait avant », les refactors qui violent une décision oubliée. L'idéal d'Axon retire cette étape en posant, AU POINT DE FUSION et non a posteriori, deux structures que la lignée intellectuelle d'Axon a forgées : le MODÈLE CAUSAL de Pearl (le pourquoi causal, le do-calculus de l'intention : ce qui change si on touche ceci) et le GRAPHE D'INTENTION de Sewall Wright / path analysis (les chemins de gouvernance vision→pilier→décision→exigence→code). Joindre ces deux structures au QUOI tri-modal (le code que FTS+vecteur+graphe RRF désignent) rend la réponse correcte PAR CONSTRUCTION : l'intention gouvernante arrive AVEC le code, donc l'agent n'a plus à la deviner. La distinction nette avec un meilleur RAG : un meilleur RAG améliore le rappel du QUOI mais laisse l'abduction du POURQUOI à la charge de l'agent — il reste faillible ; l'idéal d'Axon déplace le pourquoi DANS la récupération elle-même (why-primary, how-inferred), ce qu'aucun ajustement de moteur lexical/vectoriel ne peut atteindre. C'est aussi la valeur commerciale directe : parce que l'intention vétéran est servie avec le code, un agent fraîchement arrivé atteint INSTANTANÉMENT le niveau de quelqu'un qui connaît la base depuis des années — l'onboarding tend vers zéro et la régression par intention oubliée disparaît.

2 · Flux actuel (tel qu'il est)

La surface MCP existe et est agent-native : les outils query (symboles), inspect (détail), impact (blast radius), why (rationale SOLL), path (flux de dépendances), anomalies (risques structurels), retrieve_context + retrieve_context_v2 (récupération hybride FTS+vecteur+graphe par RRF, MIL-AXO-017 slice 4 / REQ-AXO-298) et soll_query_context (intention) sont tous exposés et machine-actionnables. Le graphe IST canonique vit en RAM = IstGraphView (snapshot CSR compact, PIL-AXO-9002) avec re-vérification PG au fetch et fallback PG explicite quand la RAM est froide — le tri-modal s'appuie donc sur le graphe RAM, pas sur des requêtes récursives PG. La réactivité MCP vient d'être RESTAURÉE cette session (le hot path télémétrie ne bloque plus le runtime), donc la surface répond. GAPS ACTUELS, au regard de la cible : (A) la FUSION EXPLICITE DU POURQUOI gouvernant AVEC le code à la récupération n'est pas encore systématique dans TOUS les outils — le why est bien récupérable via why et soll_query_context, mais il n'est pas fusionné PAR DÉFAUT dans chaque réponse de query ou inspect ; l'agent doit encore faire un second appel pour obtenir l'intention, ce qui rouvre la porte à l'abduction entre les deux appels. (B) le tri-modal RRF complet est LIVRÉ pour retrieve_context_v2 mais la couverture par-outil n'est pas uniforme — code_search lexical reste largement subsumé par retrieve_context_v2 (REQ-AXO-292 backlog) plutôt que rebâti sur la même fusion. (C) la calibration trust/freshness est exposée (CPT-AXO-029) mais le gradient de fallback reste perfectible (formulation et granularité du signal de confiance). Le socle — surface agent-native + RAM CSR + fallback PG + RRF hybride — est en place ; ce qui manque est la SYSTÉMATISATION de la fusion du pourquoi et l'UNIFORMITÉ tri-modale entre outils.

Faits vérifiés (MCP / SOLL)

  • La surface MCP agent-native expose des outils de récupération machine-actionnables : query (symboles), inspect (détail), impact (blast radius), why (rationale SOLL), path (flux de dépendances), anomalies (risques structurels), retrieve_context et retrieve_context_v2 (récupération), soll_query_context (intention).
  • retrieve_context_v2 réalise la récupération HYBRIDE tri-modale — FTS lexical + vecteur sémantique + graphe structurel — combinée par Reciprocal Rank Fusion (RRF) ; livré sous MIL-AXO-017 slice 4 / REQ-AXO-298.
  • Le vecteur sémantique repose sur pgvector avec index HNSW et des embeddings BGE-Large 1024d ; le FTS apporte la correspondance lexicale exacte ; le graphe apporte les relations structurelles réelles (appels, imports, blast radius).
  • Le graphe IST canonique est servi depuis la RAM = IstGraphView, un snapshot CSR compact (PIL-AXO-9002), avec re-vérification PG au moment du fetch ; le PG public.edge est la couche de persistance et le FALLBACK explicite quand la RAM est froide.
  • Principe directeur why-primary, how-inferred : l'intention (le pourquoi) est primaire et le comment s'en infère ; l'asymétrie est irréductible (why→how élague, how→why est lossy). Conséquence produit : fusionner le why gouvernant AVEC le code à la récupération est la raison d'être d'Axon (VIS-AXO-001).
  • CPT-AXO-029 : la fraîcheur (freshness) est une CALIBRATION DE CONFIANCE, pas un gate d'usage — les reads structurels (query/inspect/impact/why/anomalies/path) restent utilisables même quand l'indexer est idle ; on cross-vérifie selon le lag au lieu de refuser l'outil.
  • code_search (récupération lexicale / texte) est largement subsumé par retrieve_context_v2 et figure au backlog REQ-AXO-292, donc le chemin lexical pur n'est pas encore rebâti sur la fusion RRF unique.
  • Lignée intellectuelle de la vision (VIS-AXO-001) : Peirce (abduction faillible à supprimer) → Sewall Wright (graphe d'intention / path analysis) → Pearl (modèle causal / do-calculus de l'intention) — ces structures sont à poser AU POINT DE FUSION, pas inférées a posteriori par l'agent.
