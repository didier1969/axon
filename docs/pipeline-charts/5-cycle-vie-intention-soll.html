1 · Flux idéal SOTA (le nord-vrai, premiers principes)

Le nord-vrai du SOLL est la CONTINUITÉ D'INTENTION (PIL-AXO-003) : SOLL est la couche de vérité intentionnelle canonique de TOUT projet équipé d'Axon — pas seulement d'Axon lui-même. Dans l'idéal, l'intention — Vision → Piliers → Concepts → Décisions → Exigences → Validations → Stakeholders → Guidelines (les 9 types d'entités canoniques) — est CAPTURÉE à l'instant de l'observation (au moment où une friction, une décision ou une contrainte apparaît), LIÉE causalement au Concept ou au Pilier qui la gouverne, SCORÉE par impact descendant (combien d'aval dépend d'elle), PLANIFIÉE en vagues d'unblockers (ordre topologique de déblocage), puis APPLIQUÉE automatiquement au point de livraison (les guidelines pertinentes sont injectées au pre-flight, pas relues de mémoire). Cette intention SURVIT aux sessions, aux humains et aux releases. Cinq principes SOTA la structurent : (1) EVENT-SOURCING — soll.Revision est un journal d'intention immuable et rejouable (PIL-AXO-9004) : tout état SOLL dérivé (work_plan, validations, autodoc) n'est qu'une PROJECTION du journal, le rollback se fait par revision, l'audit est natif ; (2) why-primary, how-inferred — l'intention (le pourquoi) gouverne et le code (le comment) s'en infère, asymétrie directrice de tout Axon (VIS-AXO-001) ; (3) un GRAPHE D'INTENTION typé (9 types d'entités) requêtable et FUSIONNÉ à l'IST, de sorte que le why gouvernant est récupéré AVEC le code qu'il justifie ; (4) une BOUCLE AGENT-NATIVE fermée : observe la friction → log SOLL → link au gouvernant → re-plan → exécute la wave-1 → applique les guidelines à la livraison → commit ; (5) un ONBOARDING DÉTERMINISTE : un nouvel agent hérite l'intention ET la méthodologie sans ré-explication humaine. SOLL n'est donc pas un tracker de tickets : c'est la mémoire causale event-sourced qui rend l'intention persistante et rejouable, la moitié-produit indissociable de l'IST.

Pourquoi c'est l'idéal (pas un tracker de tickets) : Ce n'est pas un tracker de tickets — c'est la MÉMOIRE CAUSALE EVENT-SOURCED qui rend l'intention persistante et rejouable, la moitié-produit indissociable de l'IST. Un tracker stocke un état (ouvert/fermé) qu'on interroge ; SOLL stocke un JOURNAL D'INTENTION immuable (soll.Revision, PIL-AXO-9004) dont tout le reste — work_plan, validations, autodoc — n'est qu'une projection jetable et régénérable. Cinq sauts structurels qu'un tracker ne franchit pas : (1) Event sourcing — l'intention n'est jamais « écrasée », elle est versionnée et rejouable, ce qui donne le rollback par revision, le time-travel, l'audit post-mortem et la régénération à coût nul de toute surface dérivée ; un nouvel agent rejoue depuis l'offset zéro et hérite l'état exact de l'intention. (2) why-primary, how-inferred — l'intention gouverne le code et non l'inverse : asymétrie why→how (élagage) vs how→why (lossy), raison d'être d'Axon (VIS-AXO-001). (3) Graphe d'intention typé (9 types) FUSIONNÉ à l'IST — au moment de la récupération, le why gouvernant remonte AVEC le code qu'il justifie, ce qu'aucun tracker hors-bande ne peut faire. (4) Boucle agent-native fermée — observe → log → link → re-plan → applique guidelines à la livraison → commit : la méthodologie est exécutée par l'agent, pas relue par un humain. (5) Onboarding déterministe — l'intention ET la méthodologie s'héritent (CPT-AXO-020), supprimant la ré-explication. SOLL est ainsi la mémoire ET la méthodologie de VIS-AXO-001 : la couche intentionnelle canonique de tout projet équipé d'Axon, pas seulement d'Axon.

2 · Flux actuel (tel qu'il est)

La surface SOLL livrée couvre déjà l'essentiel du cycle. LECTURE : soll_query_context (récupération d'intention par contexte) + soll_work_plan (ordre topologique d'unblockers scoré par impact descendant ET décroissance temporelle) + soll_validate (vérification de cohérence). ÉCRITURE : soll_manager (CRUD + action=link causal) / infer_soll_mutation (proposition de mutation) / entrench_nuance (consolidation) / soll_apply_plan — les mutations Cypher INSERT/UPDATE directes sont INTERDITES, l'accès est read-only par spec, toute écriture passe par les outils. LIVRAISON : axon_pre_flight_check → axon_commit_work applique les guidelines pertinentes au point de livraison. JOURNAL : soll.Revision est en place, le rollback se fait via soll_rollback_revision. Le DDL SOLL est canonique, typé et unifié (REQ-AXO-901881, colonne title — pas label). L'autodoc HTML par projet existe (soll_generate_docs → docs/derived/soll/). Le modèle est TWO-SIDED : namespace PRO (méthodologie cross-tenant livrée au client pour SES projets) vs AXO (architecture interne d'Axon). GAPS actuels : le refresh des surfaces dérivées (autodoc notamment) est À-LA-DEMANDE, pas automatique post-mutation (REQ-AXO-309) — risque de dérive silencieuse vs le journal ; et si soll.Revision est bien un journal en place, la couverture event-sourced COMPLÈTE — où TOUTE projection (work_plan, validations, autodoc, onboarding) est 100% rejouable depuis le seul journal — reste partielle : PIL-AXO-9004 est une cible vers laquelle l'existant converge, pas un état pleinement atteint.

Faits vérifiés (MCP / code)

  • Surface SOLL — lecture : soll_query_context (récupération d'intention par contexte), soll_work_plan (ordre topologique d'unblockers scoré par impact descendant + décroissance temporelle) et soll_validate (cohérence) sont livrés et opérationnels.
  • Surface SOLL — écriture : soll_manager (CRUD + action=link causal), infer_soll_mutation, entrench_nuance et soll_apply_plan ; les mutations Cypher INSERT/UPDATE directes sont INTERDITES (read-only par spec) — toute écriture transite par les outils.
  • Surface SOLL — livraison : axon_pre_flight_check → axon_commit_work applique les guidelines pertinentes au point de livraison (les guidelines sont injectées, pas relues de mémoire).
  • 9 types d'entités canoniques : Vision · Pilier · Concept · Décision · Exigence · Validation · Stakeholder · Guideline — le graphe d'intention typé est requêtable et destiné à fusionner avec l'IST.
  • Journal Revision : soll.Revision est un journal d'intention en place ; le rollback se fait via soll_rollback_revision (jamais de suppression de nœud SOLL).
  • DDL canonique typé : le DDL SOLL est unifié et typé (REQ-AXO-901881) ; la colonne canonique est title (et non label — SELECT label casse en PG 42703).
  • Autodoc par projet : soll_generate_docs produit l'autodoc HTML par projet (docs/derived/soll/) — surface dérivée non-canonique reconstructible depuis le journal.
  • Modèle two-sided PRO / AXO : namespace PRO = méthodologie cross-tenant livrée au client pour SES projets ; namespace AXO = architecture interne d'Axon. Test : « le client reçoit-il cette règle ? » oui→PRO, non→AXO.
  • Gap refresh dérivé : le rafraîchissement des surfaces dérivées (autodoc en tête) est à-la-demande, pas automatique post-mutation (REQ-AXO-309) — risque de dérive silencieuse vs le journal.
  • Gap event-sourcing complet : soll.Revision existe, mais la couverture event-sourced totale (toute projection rejouable depuis le seul journal) reste partielle — PIL-AXO-9004 est une cible, pas un état pleinement atteint.
