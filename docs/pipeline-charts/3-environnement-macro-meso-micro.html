Trois surfaces de visualisation dérivées non-canoniques (PIL-AXO-009) : Dashboard Elixir/Phoenix, autodoc SOLL web, projection Memgraph. Le stockage est unifié deux-étages (PIL-AXO-9002) : PG OLTP canonique (soll.* + public.*) + pgvector HNSW + tsvector FTS + miroirs RAM read-only (SOLL petgraph toujours, IST CSR optionnel). Deux instances du même produit cohabitent (PIL-AXO-004) : live = vérité stable, dev = bac à sable resettable, topologie strictement isolée (ports/sockets/pidfiles/DB distincts) avec priorité asymétrique sous contention (live > dev).

Couches macro · meso · micro

MACRO — Identité produit two-sided + dual sous-produit (PIL-AXO-9003, PIL-AXO-008)

Axon-produit (plateforme commerciale, méthodologie PRO cross-tenant)Axon-projet (codebase AXO, tenant zéro dogfood)Sous-produit Brain (axon-brain, surface MCP)Sous-produit Indexer (axon-indexer, IST writer)Méthodologie cross-tenant (namespace PRO) comme 3e axe commercial séparable
Axon est opéré sous deux identités simultanées (PIL-AXO-9003) dont la confusion est la 1re cause de drift de placement : Axon-produit vendu aux entreprises (runtime + méthodologie embarquée, namespace PRO = brochure commerciale queryable) et Axon-projet (codebase /home/dstadel/projects/axon, namespace AXO = scripts/build/qualify/promote/tests). Métacircularité dogfood : Axon-projet = tenant zéro parmi N+1. Le produit se décompose en deux sous-produits binaires (PIL-AXO-008) activables isolément : Brain (owns soll.*, expose MCP HTTP, 60 outils) et Indexer (owns public.* IST, aucune surface MCP). Cas de déploiement légitimes : Brain seul (consommer KG préconstruit), Indexer seul (backfill/reindex), Brain+Indexer même hôte (standard, régi par PIL-004), Brain+Indexer hôtes séparés (scale-out futur). AxonRuntimeMode (runtime.rs) = unique source de vérité d_activation.

Stockage deux-étages (PIL-AXO-9002) : PG seul écrivain canonique (soll.* possédé par Brain, public.* IST possédé par Indexer), miroirs RAM read-only dérivés avec re-vérification PG au fetch (aucun fantôme atteint le LLM) et fallback gradienté sur freshness_lag_ms. AGE retiré (pg_extension : vector + pg_trgm seulement, pas age). Le superviseur process-compose pilote un DAG déclaratif (postgres-check -> brain -> dashboard) avec readiness/liveness probes HTTP, restart on_failure, shutdown ordonné. Trois surfaces de visualisation humaine (PIL-AXO-009) read-only 100% reconstructibles depuis PG : Dashboard Phoenix LiveView (src/dashboard/), autodoc SOLL HTML (docs/derived/soll/ via soll_generate_docs), Memgraph (scripts/memgraph-projection.sh). Aucune n_est autoritative.
