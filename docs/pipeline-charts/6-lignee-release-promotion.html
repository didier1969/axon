1 · Flux idéal SOTA (le nord-vrai, premiers principes)

Le nord-vrai de la release est la LIGNÉE DE RELEASE QUALIFIÉE (PIL-AXO-005) généralisée vers un CONTROL-PLANE DÉCLARATIF multi-environnement (PIL-AXO-9005). La promotion vers live n'est jamais inférée d'un push git : elle est EXPLICITE, artefact-driven, versionnée et vérifiée contre l'identité réelle du runtime en cours. Cinq principes structurels. (1) On ne promeut QUE du hash-vérifiable : artefacts reproductibles épinglés par empreinte SHA-256 (GUI-PRO-006, builds déterministes), identiques partout — promotable même en air-gap parce que l'identité de l'artefact ne dépend pas d'un réseau ni d'un horodatage de build. (2) Séquence qualifiée stricte : un manifeste épingle le candidat → le candidat est QUALIFIÉ avant tout swap (preflight + qualify-mcp couvrant quality ET latency) → swap ATOMIQUE des binaires → on VÉRIFIE que le runtime réellement en cours == l'identité du manifeste (pas une supposition, une mesure) → qualify final post-swap. (3) Zéro-downtime et fail-closed : si le HEAD bouge pendant la séquence, on n'applique rien (la promotion devient un instantané cohérent ou n'a pas lieu). (4) Rollback par manifeste historique : on revient à un état déjà-qualifié connu, jamais via git revert (qui re-construirait du non-qualifié). (5) CIBLE control-plane déclaratif (PIL-AXO-9005) : chaque environnement — tenant × stage × région — est DÉCRIT en IaC ; un réconciliateur compare en continu l'état désiré (manifeste IaC) à l'état observé (runtime mesuré) ; toute dérive IaC↔runtime devient une anomalie de PREMIÈRE classe (détectée, alertée, réconciliée) et non un drift silencieux ; la topologie est self-describing — un seul appel rend une carte vivante de tous les environnements et de leur identité courante (MIL-AXO-027 slice-8).

Pourquoi c'est l'idéal (pas un script de déploiement amélioré) : Ce n'est pas une version plus propre de promote_live_safe.sh ; c'est le pattern Kubernetes/GitOps — RÉCONCILIATION desired-vs-observed + SUPPLY-CHAIN reproductible — appliqué à la connaissance structurelle. La distinction de fond avec un meilleur script : un script, aussi soigné soit-il, reste IMPÉRATIF (« fais ces étapes »), single-host, et croit sa propre sortie ; l'idéal est DÉCLARATIF (« voici l'état désiré de chaque environnement ») et VÉRIFIE l'identité réelle du runtime au lieu de la supposer. Quatre sauts qu'un script ne peut pas franchir en se nettoyant. (1) On ne place que des artefacts HASH-VÉRIFIABLES (SHA-256, GUI-PRO-006) : l'identité d'une release est son empreinte reproductible, pas un numéro de build ni un horodatage ; identique partout, promotable en air-gap, comparable bit-à-bit — c'est la base supply-chain SLSA/GitOps. (2) La promotion est EXPLICITE et artefact-driven, jamais inférée d'un push git : un manifeste épingle le candidat, et c'est le manifeste — pas l'historique git — qui est la source de vérité de ce qui doit tourner. (3) On VÉRIFIE que le runtime en cours == l'identité du manifeste : la promotion n'est « réussie » que si la mesure le confirme, ce qui supprime la classe entière de bugs « j'ai copié le binaire donc c'est bon » et de faux succès/échecs de finalisation. (4) Toute DÉRIVE IaC↔runtime est une anomalie de première classe : dans un control-plane déclaratif, l'écart entre désiré et observé est détecté et réconcilié en continu (GUI-PRO-012, IaC/GitOps), au lieu de dériver silencieusement entre deux promotions manuelles — et la topologie est self-describing (un appel = carte vivante de tenant × stage × région, MIL-AXO-027 slice-8) plutôt que reconstruite à la main host par host.

2 · Flux actuel (tel qu'il est)

Le chemin de promotion canonique EXISTE et fonctionne : scripts/release/promote_live_safe.sh --project AXO exécute en one-shot la lignée — rebuild du candidat → release-preflight → création d'un manifeste épinglé → swap des binaires dans bin/ + restart live → match de l'identité runtime MCP contre le manifeste → qualify-mcp final (quality + latency) avant de DÉCLARER la promotion. L'état est matérialisé déclarativement dans un manifeste .axon/live-release/current.json (build_id, generation-id, state=promoted) avec un historique versionné sous .axon/live-release/history/. Le rollback est par manifeste : scripts/release/rollback_live.sh restaure un manifeste historique (jamais git revert). Les binaires live sont TOUJOURS installés par la pipeline dans bin/ — jamais cargo build --release manuel — et qualify-mcp valide quality+latency avant de déclarer la promotion saine. C'est déjà la lignée qualifiée single-host de PIL-AXO-005. GAPS actuels. (1) SMELLS de script trompeurs : le step-7 « finalize » de promote_live_safe ÉCHOUE faussement sur un check « no pid file » et émet un faux warning « binary md5 unchanged after promote » alors que la promotion est en fait RÉUSSIE (manifeste promu, qualify-mcp ok, binaire en place, runtime sain) — le script ment sur son propre succès/échec, exactement la classe de bug que l'idéal supprime en vérifiant l'identité réelle au lieu de la deviner. (2) La CIBLE control-plane déclaratif + topologie self-describing multi-tenant (PIL-AXO-9005, MIL-AXO-027 slice-8) n'est PAS atteinte : aujourd'hui = promotion scriptée single-host, pas de réconciliation desired-vs-observed continue, pas de détection de dérive IaC↔runtime de première classe, pas de carte vivante un-appel des environnements tenant × stage × région. (3) Séparation d'artefact aplatie : l'artefact dev EST l'artefact live promu (même binaire release) — il n'y a pas de frontière d'artefact distincte entre ce qui est testé en dev et ce qui est promu en live.

Faits vérifiés (code / SOLL)

  • Le chemin de promotion canonique est scripts/release/promote_live_safe.sh --project AXO : one-shot rebuild candidat → release-preflight → création d'un manifeste épinglé → swap des binaires dans bin/ + restart live → match de l'identité runtime MCP contre le manifeste → qualify-mcp final (quality + latency) avant de déclarer la promotion.
  • L'état de release est matérialisé dans un manifeste .axon/live-release/current.json (build_id, generation-id, state=promoted) avec un historique versionné sous .axon/live-release/history/.
  • Le rollback est par manifeste : scripts/release/rollback_live.sh restaure un manifeste historique déjà-qualifié — jamais via git revert (qui re-construirait du non-qualifié).
  • Les binaires live sont TOUJOURS installés par la pipeline dans bin/, jamais par cargo build --release manuel suivi d'une copie ; qualify-mcp valide quality+latency avant que la promotion soit déclarée saine.
  • SMELL observé cette session — le step-7 « finalize » de promote_live_safe ÉCHOUE faussement sur un check « no pid file » et émet un faux warning « binary md5 unchanged after promote » alors que la promotion est en réalité RÉUSSIE : manifeste promu, qualify-mcp ok, binaire en place, runtime sain. Le script ment sur son propre succès/échec — c'est exactement la classe de bug que l'idéal supprime en jugeant sur l'identité runtime vérifiée plutôt que sur des proxies (pid file, md5 brut).
  • La CIBLE control-plane déclaratif multi-tenant (PIL-AXO-9005) et la topologie self-describing un-appel (MIL-AXO-027 slice-8) ne sont PAS atteintes : aujourd'hui la promotion est scriptée et single-host, sans réconciliation continue desired-vs-observed, sans détection de dérive IaC↔runtime de première classe, sans carte vivante des environnements tenant × stage × région.
  • La séparation d'artefact dev/live est aplatie : l'artefact dev est l'artefact live promu (même binaire release), il n'existe pas de frontière d'artefact distincte entre ce qui est testé en dev et ce qui est promu en live.
  • La promotion est déjà EXPLICITE et artefact-driven (manifeste épinglé), conforme à PIL-AXO-005 : elle n'est jamais inférée d'un push git — c'est l'acquis solide sur lequel la généralisation déclarative (PIL-AXO-9005) doit s'appuyer.
