Structure SOLL : Schéma des Relations

La couche intentionnelle (SOLL) définit l'ontologie du projet. Elle est isolée dans soll.db et structure la connaissance selon une hiérarchie stricte certifiée par le Registre.

Strate Macro (Vision & Piliers)

La Vision définit l'objectif ultime. Elle est soutenue par des Pillars (Piliers) qui représentent les domaines de contraintes majeures du système.

Strate Meso (Requirements)

Les Requirements (Exigences) décomposent les piliers en objectifs fonctionnels vérifiables, identifiés par un DNA unique (ex: REQ-PROJ-001).

Strate Micro (Concepts)

Les Concepts fournissent le "Comment" technique. Ils expliquent les exigences et sont physiquement liés au code source (IST).

Architecture Sanctuary (v3.3)

Isolation physique des intentions humaines pour garantir l'immunité face aux réinitialisations techniques.

Le Sanctuaire (soll.db)

Contient la Vision, les Piliers, les Requirements et le Registre d'IDs. Ce fichier est sacré et ne subit jamais de purge automatique.

La Forge (ist.db)

Contient les données volatiles : fichiers indexés, symboles AST et dépendances physiques. C'est la cible exclusive de la commande reset.

Mécanisme ATTACH

Au démarrage, Axon ouvre ist.db et attache soll.db sous un namespace dédié. Les requêtes Cypher peuvent croiser les données entre les deux mondes en toute transparence.

"Grâce au Sanctuaire, même si nous reconstruisons toute la forge technique (IST), je n'oublie jamais vos exigences stratégiques (SOLL)."

Gestion de la Concurrence (MVCC vs Guardian)

Pourquoi le LLM accède en direct ?

Le serveur MCP interroge DuckDB via une connexion Read-Only dédiée. Grâce au MVCC, il accède à un instantané (Snapshot). Cela évite de charger l'orchestrateur Elixir et garantit une latence < 100ms.

Le rôle du Guardian

Le Guardian ne gère pas les lectures. Il est le Contrôleur de Débit pour les écritures. Il décide qui travaille et quand, s'assurant qu'aucun worker ne sature les I/O de la base de données.

Le Writer Actor

En Rust, toutes les écritures passent par un Acteur Unique. C'est le seul point d'entrée pour les transactions `WRITE`, éliminant physiquement les erreurs `Database busy`.

Scalabilité et Multi-Agents

Axon est conçu pour servir simultanément plusieurs instances d'agents IA (Claude, Gemini, etc.) sans dégradation de performance.

Multiplexage Tokio

Le serveur MCP utilise le runtime asynchrone Tokio. Il peut gérer des centaines de connexions concurrentes sur un pool de threads réduit, évitant ainsi le blocage I/O classique.

Multi-Reader MVCC

DuckDB permet des lectures parallèles illimitées. Chaque agent IA dispose de sa propre vue cohérente du graphe, isolée des écritures massives de l'ingestion.

Réserve de Cycles MCP

Une portion des ressources CPU (Core Reserve) est sanctuarisée pour le serveur MCP. Même sous une charge d'indexation de 100%, l'intelligence reste réactive.

Compliance & Traçabilité (MBSE)

Axon implémente l'ingénierie basée sur les modèles (Model-Based Systems Engineering) pour assurer un "Digital Thread" entre l'intention et le code.

Traçabilité Bidirectionnelle

Chaque ligne de code est reliée à un Concept (SOLL). Le LLM peut justifier chaque fonction par rapport à un Requirement original de la Roadmap.

Généalogie & Historique

La relation SUPERSEDES permet de faire évoluer le système sans perdre la mémoire. Un nouveau concept "écrase" l'ancien mais conserve un lien vers lui pour expliquer le pivot.

Calcul d'Avancement Réel

L'état "Terminé" est une preuve physique. Si un Requirement n'est pas relié à un Symbol (Code) via un Concept, il est marqué comme En attente.

Souveraineté des IDs

Le système génère les identifiants (ex: REQ-AXO-004). Cette autorité centrale garantit l'unicité du Digital Thread à travers tout le graphe.

Écosystème Technologique

Un socle polyglotte haute performance, hermétique et reproductible.

Control Plane Elixir / OTP

Orchestration résiliente basée sur le modèle d'acteurs. Gestion granulaire des cœurs CPU et du traffic shaping.

Data Plane Rust / Tokio

Calcul intensif (Tree-sitter, FastEmbed ONNX). Isolation totale des ressources et performance brute.

Sanctuary Storage DuckDB (Dual-DB)

Isolation physique SOLL/IST via ATTACH DATABASE. Source unique pour l'intention et la réalité physique.

Environment Nix / Devenv

Isolation hermétique de la chaîne de compilation et des dépendances. Reproductibilité 100% garantie.

Traffic Shaping & Protocole Titan

Axon utilise les métadonnées du graphe pour arbitrer l'accès aux ressources CPU.

Priorisation Dynamique

Chaque nœud File possède une propriété priority (0-100). Le Guardian "tire" en priorité les fichiers avec le score le plus élevé (fichiers ouverts ou requis par le LLM).

Le Protocole Titan

Les fichiers > 1MB sont marqués dès le scan. Ils sont soit traités en mode "structure seule" (sans embeddings), soit isolés pour éviter de saturer la RAM des autres workers.

Zéro Oubli Sémantique

Même un fichier rejeté reste présent dans DuckDB avec le statut titan_ignored. Le LLM peut ainsi justifier ses lacunes d'analyse auprès de l'utilisateur.

Utilisation Optimale des Cœurs (Pipelining)

Pour atteindre 100% d'utilisation CPU, Axon implémente un mécanisme de "Double-Buffering" entre Elixir et Rust.

La Trémie ETS

Le Guardian ne polle pas DuckDB pour chaque fichier. Il remplit une table ETS en RAM par lots de 100. Cette taille réduite garantit que les fichiers prioritaires ne sont pas bloqués par un backlog trop important.

Observabilité : Tracer

Le système utilise Axon.Watcher.Tracer pour mesurer les micro-latences (T0 à T4). Ces mesures permettent au Guardian d'ajuster le débit en temps réel pour éviter la saturation.

Points de Mesure (:telemetry)

Le système surveille le fill_rate de la trémie ETS et la pressure des sockets. Si un goulot apparaît, le Guardian ajuste dynamiquement la taille des lots de pull.

Régulation de Flux Adaptative (v3.2)

Axon ajuste dynamiquement sa "matière première" en fonction de la vitesse réelle de traitement pour garantir une latence déterministe.

Calcul de Fenêtre Glissante

Le Guardian calcule la taille optimale du buffer : Fenêtre = (Débit Files/sec) * Latence_Cible. Si le système ralentit, le buffer rétrécit pour laisser passer les urgences plus vite.

Seuils Watermark (Min/Max)

Le système évite les temps morts avec un rechargement anticipé. Si la cible est de 200 fichiers, le Guardian déclenche un nouvel import dès que le buffer descend à 100.

Priorité Sémantique

Lors de chaque rechargement, les fichiers sont extraits par ORDER BY priority DESC. Un fichier urgent n'attendra jamais plus d'un cycle de fenêtre (max 10s).

Principes d'Ingénierie Nexus

Zero-Sleep

Aucune attente arbitraire. Le système est auto-régulé par la disponibilité physique des cœurs CPU.

Backpressure Physique

Le flux est tiré (Pull). On n'insère un nouveau fichier que si un slot est libre en base et sur le CPU.

Atomicité

Chaque fichier est traité comme une transaction atomique. Pas d'état "partiellement indexé".
