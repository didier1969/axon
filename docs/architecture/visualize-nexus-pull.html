Axon Pipeline v2 streaming-v2
Six stages, deux lanes, zéro machine-à-états

Architecture canonicalisée par CPT-AXO-053 + CPT-AXO-054. Le watcher pousse les fichiers nouveaux/modifiés dans deux lanes indépendantes en débit : A = graphe (priorité vitesse, structure + relations + métadonnées) et B = vectorisation + lexical (chunking, embedding GPU, FTS auto). Bounded tokio::sync::mpsc entre tous les stages, idempotence par UPSERT, persistance minimale (IndexedFile 3 colonnes), et un seul try_send non-bloquant pour le cross-pipeline A1→B1 — le graphe ne ralentit JAMAIS pour le GPU.

1 · Topologie de flux six stages · bounded channels

A1 lit le fichier une fois et émet un PreparedFile qui voyage en parallèle vers A2 (parsing+graphe) et B1 (chunking+vectorisation). Le canal A1→B1 est non-bloquant avec un buffer de 10 000 fichiers — le graphe garde sa cadence quoi qu'il arrive.

3 · Propriétés architecturales six garanties graven dans le code

Chaque propriété est une décision verrouillée par construction — pas un best-effort, pas une convention, mais une invariante reflétée dans la signature des stages et la topologie des canaux.

P1Graph priority absolue

A1 fait deux choses : send().await bloquant vers A2 (lane graphe), et try_send non-bloquant vers B1 (lane vecteur). Si B1 est saturé (buffer 10 000 plein), le try_send échoue silencieusement et A1 continue à servir A2 sans attendre. Le graphe atteint sa vitesse native indépendamment de la cadence GPU.

P2Idempotence par UPSERT

A3 UPSERT (ON CONFLICT DO UPDATE) Symbol + IndexedFile, et ON CONFLICT DO NOTHING sur les relations. B1 UPSERT Chunk. B3 UPSERT ChunkEmbedding. Crash à n'importe quel stage → restart → watcher rediscover → UPSERT = no-op. Zéro orphan à reaper, zéro claim à libérer.

P3Dual-parse déterministe

A2 et B1 parsent tous deux le même content via tree-sitter (déterministe), produisent les mêmes Symbols, et dérivent les mêmes Chunks via build_symbol_chunks (fonction pure). Coût ≈ 5 ms par parse × 2 = négligeable vs GPU embed ≈ 250-280 ms / batch.

Les IDs (symbol_id, chunk_id) sont pures fonctions de (project_code, path, name, part_index) → 1:1 identiques entre A et B. Même si l'ordre des Vec divergeait, l'UPSERT garantit l'état final.

P4FTS gratuit (REQ-AXO-292)

PG FTS lexical = colonne content_tsv GENERATED ALWAYS AS … STORED sur public.Chunk. Dès que B1 fait son UPSERT Chunk, PG calcule le tsvector et l'index GIN se met à jour sans aucun code applicatif. Pas de stage C1, pas de worker FTS — l'effet est latent dans le moteur PG.

P5Multi-worker safe

Chaque stage = N workers en compétition sur un Arc<Mutex<Receiver>>. Le lock n'est tenu que pendant recv() (µs), pas pendant work() (ms-s). mpsc garantit qu'un item n'est reçu qu'une fois → aucun doublon possible, load-balance naturel, scale linéaire jusqu'à 16+ workers.

P6Activation orthogonale

AXON_RUNTIME_MODE = brain_only / indexer_graph / indexer_vector / indexer_full. Les pipelines A et B peuvent tourner indépendamment : graphe seul (pas de GPU), vecteur seul (rattrapage des chunks non-embeddés), brain seul (MCP serveur sur la base), ou les deux. Permis par MIL-AXO-015 (PG MVCC multi-writer natif).

4 · Matrice d'activation AXON_RUNTIME_MODE

Quels workers tournent selon le mode runtime sélectionné. Les modes sont mutuellement exclusifs côté binaire indexer ; le brain tourne en parallèle dans son propre process.

brain_only

A1A2A3 B1B2B3

indexer_graph

A1A2A3 B1B2B3

indexer_vector

A1A2A3 B1B2B3

indexer_full

A1A2A3 B1B2B3
Mode indexer_vector. Quand seul B tourne (graphe déjà peuplé), B1 ne reçoit RIEN du canal A1→B1 (A est inactif). Il s'appuie alors sur son cold-start poll DB : SELECT id FROM Chunk LEFT JOIN ChunkEmbedding WHERE NULL LIMIT 256 pour rattraper tous les chunks non-embeddés. Même chemin si indexer_full démarre après un long brain-only.
