Axon Canonical File Lifecycle

This document defines the real canonical flow of a file inside Axon after the 2026-04-03 lifecycle refactor. It separates structural truth from vector truth and separates normal file flow from subtree-driven rescans.

1. Normal File Path

Disk A source file exists under a workspace project root.
↓
Watcher / Scanner The file is observed directly or discovered by a scan.
↓
Ingress Guard + Buffer Duplicates and unchanged metadata are reduced before canonical writes.
↓
Promoter The file becomes canonical in File with file_stage=promoted.
↓
Claim Scheduler claims the file with file_stage=claimed.
↓
Worker → Writer The file enters writer_pending_commit until structural commit lands.
↓
Graph Indexed Structural truth is committed with graph_ready=true.

2. Directory Event Path

Directory Event A directory change is observed.
↓
Eligibility Filters Early ignore rules and blocked subtree segments decide whether the directory is even allowed to request a rescan.
↓
Subtree Hint A bounded control signal is stored in the ingress layer. It is not per-file truth.
↓
Controlled Rescan The promoter triggers a subtree scan which emits new file events.
↓
Back To Normal File Path Only emitted file events become canonical file rows.

3. Vector Path

Graph Ready Structural truth exists and is queryable.
↓
Chunk Embeddings Chunks derived from symbols are embedded asynchronously.
↓
Vector Ready vector_ready=true means the current chunk set for the file is embedded for the active model.

Structural completion is canonical. Vector completion is derived and may lag without invalidating structural truth.

4. Canonical Per-File Indicators

  • status remains compatibility-facing and operational.
  • file_stage is the canonical lifecycle stage.
  • graph_ready states structural completion.
  • vector_ready states vector completion for the current chunk set.
  • status_reason remains the human-operational cause.
